## Git: Машина времени - GitHub/GitLab workflow

В этом уроке мы разберем, как использовать Git вместе с платформами GitHub или GitLab для эффективной командной работы над проектами. Мы рассмотрим распространенный рабочий процесс, который поможет вам избежать конфликтов и поддерживать код в чистоте.

### Что такое GitHub/GitLab workflow?

GitHub/GitLab workflow - это набор практик и соглашений, определяющих, как разработчики взаимодействуют с Git-репозиторием, размещенным на GitHub или GitLab. Он помогает командам эффективно сотрудничать, управлять изменениями в коде и поддерживать стабильную кодовую базу. Основная идея workflow - работа над фичами в отдельных ветках и их последующее объединение с основной веткой (обычно `main` или `master`) через механизм запросов на слияние (pull requests или merge requests).

### Основные этапы workflow

1.  **Создание ветки:** Начните с создания новой ветки (branch) от основной ветки (`main` или `master`) для каждой новой фичи, исправления бага или эксперимента. Это изолирует ваши изменения от основного кода.

    ```bash
    git checkout -b feature/new-feature
    ```

    Этот код создаст и переключит вас на ветку `feature/new-feature`.

2.  **Работа над фичей:** Внесите необходимые изменения в код в своей ветке, регулярно коммитя изменения.

    ```bash
    git add .
    git commit -m "Implement new feature"
    ```

3.  **Публикация ветки:** Загрузите вашу ветку на GitHub/GitLab.

    ```bash
    git push origin feature/new-feature
    ```

4.  **Создание Pull Request/Merge Request:** На GitHub/GitLab создайте Pull Request (PR) или Merge Request (MR) из вашей ветки в основную ветку. Это уведомляет других разработчиков о ваших изменениях и просит их проверить код.

5.  **Code Review:** Другие разработчики просматривают ваш код, оставляют комментарии и предлагают улучшения.

6.  **Внесение изменений:** На основе отзывов внесите необходимые изменения в свою ветку и отправьте новые коммиты.  Эти коммиты автоматически появятся в PR/MR.

7.  **Слияние ветки:** После одобрения PR/MR, ветку можно слить (merge) с основной веткой.

    ```bash
    git checkout main
    git pull origin main
    git merge feature/new-feature
    git push origin main
    git branch -d feature/new-feature
    ```

    Или, проще, через интерфейс GitHub/GitLab, нажав кнопку "Merge pull request" или "Accept merge request".

8.  **Удаление ветки:** После слияния ветку можно удалить как локально, так и удаленно.

    ```bash
    git branch -d feature/new-feature # Удаление локальной ветки
    git push origin --delete feature/new-feature # Удаление удаленной ветки
    ```

### Жизненный пример

Представьте, что вы работаете над веб-приложением.  Ваша команда использует GitHub.  Ваша задача - добавить новую кнопку "Подписаться" на главной странице.

1.  Вы создаете ветку `feature/subscribe-button`.
2.  Вы добавляете HTML-код кнопки в `index.html`:

    ```html
    <button id="subscribe-button">Подписаться</button>
    ```

3.  Вы добавляете немного JavaScript для обработки клика по кнопке:

    ```javascript
    document.getElementById('subscribe-button').addEventListener('click', function() {
      alert('Вы подписались!');
    });
    ```

4.  Вы коммитите и пушите свою ветку на GitHub.
5.  Вы создаете Pull Request.
6.  Другой разработчик проверяет ваш код, замечает, что нет стилей для кнопки и предлагает добавить CSS.
7.  Вы добавляете CSS стили в `style.css`:

    ```css
    #subscribe-button {
      background-color: green;
      color: white;
      padding: 10px 20px;
      border: none;
      cursor: pointer;
    }
    ```

8.  Вы коммитите и пушите изменения. PR автоматически обновляется.
9.  После повторной проверки и одобрения, PR сливается с `main`.

### Ключевые моменты

*   **Всегда создавайте ветки для новых фич или исправлений.**
*   **Регулярно коммитьте свои изменения.**
*   **Используйте Pull Requests/Merge Requests для code review.**
*   **Будьте внимательны к отзывам и вносите необходимые изменения.**
*   **Удаляйте ветки после слияния.**
*   **Регулярно обновляйте свою локальную основную ветку (`main` или `master`) с удаленным репозиторием (`git pull origin main`).**
